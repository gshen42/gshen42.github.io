Freer Monads

Freer monads are a generalization of monads by not carrying out effects immediately but just recording/accumulating them, allowing us to separate monad implementations from their interfaces. We can define a monad and its operations (interface), write programs that use them, and then separately define their implementations. In this note, we'll use the clasic State monad to demostrate the usage of the freer monad.

State monads reprsesent computations with effects of reading and writing a single mutable state. It is commonly defined in Haskell as follows: 


            newtype State s a = State { runState :: s -> (a, s) }

            get :: State s s
            get = State $ \s -> (s, s)

            put :: s -> State s ()
            put s = State $ \_ -> ((), s)

            instance Monad (State s) where
                m >>= f = State $ \s ->
                let
                    (a, s') = runState m s
                in
                    f a

State is the state monad; it takes a pre-state and returns a tuple of the result of the computation and the post-state. Operations get and put, as their name suggest, read and write the state respectively. The monad instance simply threads the states together.

However, State is not the only way to implement the state monad. We can also define it as IO computation with access to a reference: 


            newtype State' s a = State' { runState' :: IORef s -> IO a }

            get' :: State' s s
            get' = State' $ \ref -> readIORef ref

            put' :: s -> State' s ()
            put' s = State' $ \ref -> writeIORef ref s

            instance Monad (State' s) where
                m >>= f = State' $ \ref ->
                runState' m ref >>= \a ->
                runState' (f a) ref

Although State' shares the same interface as State (expect runState'), because it is defined as a separate entity, we can't easily switch them. The user of the state monad has to have a clear mind of which implementation to use before writing any code. This is not ideal as it makes the code less flexible.

To solve this problem, we introduce the concept of freer monads. Ulike monads we defined above, which eagerly defines how they are interpreted can carries out effects immediately along the computation, freer monads interpret effectful computation algebraically as a tree and effects are carried out by adding nodes to the tree. Here is how freer monad Freer is deifned: 


            data Freer f a where
                Pure :: a -> Freer f a
                Do   :: f a -> (a -> Freer f b) -> Freer f b

Freer is parameterized by a functor f that defines the supported effects with f a represents an operation returns a value of type a. Pure represnets pure computations; Do represents effectful computatiton with an operation and a continuation.

Now the state monad can be defined as: 


            data StateF s a where
                Get :: StateF s s
                Put :: s -> StateF s ()

                type State'' s = Freer (StateF s)

                get'' :: State'' s s
                get'' = Do Get Pure

                put''' :: s -> State'' s ()
                put = Do . Put

StateF defines the allowed operations in the state monad without committing to a concrete implementation. We can then write mutiple interprer functions that correspond to different implementations of the state monad: 


            interpAsFun :: State'' s a -> s -> (a, s)
            interpAsFun (Pure a)       = \s -> (a, s)
            interpAsFun (Do Get k)     = \s -> interpAsFun (k s) s
            interpAsFun (Do (Put s) k) = \_ -> interpAsFun (k ()) s

            interpAsIO :: State'' s a -> IORef s -> IO a
            interpAsIO (Pure a)       = \_ -> return a
            interpAsIO (Do Get k)     = \ref -> readIORef ref >>= \s -> interpAsIO (k s) ref
            interpAsIO (Do (Put s) k) = \ref -> writeIORef ref s >> interpAsIO (k ()) ref
